### 问题 13：LLM 生成板块定义的降级策略

**提问**：LLM 不是 100% 可靠。四大风险点：
1. JSON 格式错误
2. Topic 幻觉（编造不存在的 topic）
3. 板块过于宽泛或过窄
4. 中文/英文 topic 混用

**原始方案（三层防御）**：
1. **Prompt 强约束** — 限制 topic 来源、禁止生造词
2. **JSON 修复 + Topic 校验** — `json_repair` + SQLite 校验真实 topic
3. **通用模板兜底** — 降级到基础 6 板块

**用户深入追问**：这三层防御够吗？要不要加一步用户确认的交互？

**系统补充分析**：
- **不建议阻断式确认**：日报系统应"即点即看"？阻断交互破坏自动化感
- **第一层补强**：增加 Few-Shot 示例 + Negative Prompt + `response_format: json_object`
- **第二层补强**：无效 Topic **不丢弃**？降级转为 `match_keywords` 做文本搜索
- **第三层补强**：用动态元模板拼接（`{user_topic} Core` / `{user_topic} Tools` / `{user_topic} Research`）
- **交互策略**：首屏直出 → 右上角 `🔧 自定义板块` 编辑入口 → 用户改过后持久化 → 下次直接读 DB

**最终 4 层防御体系**：

| 层级 | 原方案 | 进化版补充 | 是否采纳 |
|------|--------|-----------|:--:|
| 1 层 | Prompt 约束规则 | + Few-Shot + Negative Prompt + `json_object` 模式 | ✅ |
| 1.5 层 | 无 | DeepSeek `response_format: json_object` | ✅ |
| 2 层 | `json_repair` + 丢弃无效 topic | 无效 topic 降级转为 `match_keywords` | ✅ |
| 3 层 | 通用 6 模板 | 动态元模板拼接 | ✅ |
| 交互 | 阻断确认 | 首屏直出 + 编辑入口 + 持久化 | ✅ |

**用户回答**：你的补充非常精准。逐条确认全部采纳。

### 问题 18：Web 技术栈选型

**提问**：Flask vs FastAPI vs 纯 http.server？

**分析**：
- Flask + Jinja2 原生绝配（项目已有 jinja2 依赖）
- FastAPI 强在 JSON API 而非 SSR
- 当前依赖 `requests` 是同步阻塞的？用 async 没有性能提升反而增加复杂度

**建议**：Flask + Jinja2 + 前端 `fetch` 异步刷新 + `lru_cache` 缓存 LLM 结果。

**用户回答**：结论非常明确？坚定地选方案 A（Flask + Jinja2）。并深入补充了 Flask 优于 FastAPI 的理由和 3 个防坑指南。

## 总结

经过 **19 轮结构化盘问**，我们从"当前评分体系太常规"这个模糊的问题出发，逐步走完了从**问题诊断 → 数据源扩展 → 多维度评分公式设计 → 6 板块日报结构 → 专业类/基础类分层 → 自定义模块（自然语言输入 → LLM 动态板块生成 → 4 层防御降级）→ Web 交互 → 架构重构 → 技术选型 → 实施优先级**的完整决策链。

### 核心设计亮点

1. **多维度评价体系**：从单一 Star 排序升级为 6 维度（健康度、活跃度、成熟度、增长趋势、社区参与、AI 生态匹配），全部数据来自 GitHub REST API 免费端点，零外部依赖。

2. **故事线板块结构**：不同于传统 Trending 列表的"一刀切"，每个项目根据其特质落到最匹配的板块（正在爆发 / 质量标杆 / 潜力新星 / 热度陷阱），每个板块有自己的准入条件和排序公式。

3. **LLM 动态板块生成**：用户输入任意话题（如"量化交易"）→ DeepSeek 生成该垂类的专属板块定义（JSON）→ 系统据此筛选排序渲染。这是整个项目最具创造性的 feature。

4. **4 层降级防御体系**：`json_object` 模式确保格式正确 → Few-Shot 防止幻觉 → 无效 Topic 降级转关键词检索 → 动态元模板兜底（让降级后的日报依然有话题感）。

5. **产品级交付体验**：Flask Web 服务 + 模式 TAB 动态切换 + 前端异步刷新 + HTML 导出 + 右上角板块编辑入口。评委可以直接在浏览器中体验完整交互。

按照实施优先级顺序，从第一批（架构重构 + 基础日报升级 + Flask Web）开始编码实现。
